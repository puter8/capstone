## 2. 핵심 질문 TOP 6 — 답변

### Q1. 콜드 스타트 패러독스
> "관계 기반 유대감은 시간이 지나야 형성되는 특성상, 초반에는 기존 앱과 차별점을 체감하기 어려울 수 있습니다. 78% 이탈이 3개월 안에 발생한다면, 아직 기억이 쌓이지 않은 초기 세션에서 사용자를 어떻게 붙잡으실 계획인가요?"

**답변:**
초기 세션에서는 EMA의 스무딩 계수 α를 일시적으로 높게 설정해(예: 0.3 → 0.7) 캐릭터 변화를 의도적으로 빠르게 만든다. 즉, 첫 3~5턴 안에 사용자가 "내 말투에 반응하고 있다"는 것을 체감하게 하는 것이 목표다.

구체적으로는 두 가지 장치를 둔다.
- **온보딩 퀵 설정**: 첫 진입 시 "격식체 / 캐주얼 / 탐구형" 중 선호 스타일을 선택하게 해 초기 5축 값을 사전 설정한다. 이렇게 하면 첫 대화부터 이미 어느 정도 맞춰진 캐릭터로 시작할 수 있다.
- **시각적 피드백**: 캐릭터 형태가 실시간으로 변하는 Canvas2D 애니메이션을 초기부터 보여줌으로써, 수치 변화가 눈에 보이는 형태로 즉각 전달되도록 한다.

유대감은 시간이 걸리지만, "내가 영향을 주고 있다"는 느낌은 첫 세션에서 줄 수 있다.

### Q2. 캐릭터 정체성 제어 로직
> "캐릭터의 '핵심 자아'를 유지하면서도 '관계적 변화'만 허용하는 제어 로직이 있나요?"

**답변:**
5축 자체를 두 레이어로 구분해 관리한다.

- **기저 성격 (Base Personality)**: 캐릭터 선택 시 고정되는 초기값. 예를 들어 "학구적인 캐릭터"는 Curiosity 기저값이 70으로 고정된다. 이 값은 EMA로 변하지 않는다.
- **관계 레이어 (Relationship Layer)**: 사용자와의 대화로 축적되는 Intimacy와 Formality 편차. 이 부분만 EMA로 업데이트된다.

기술적으로는 CHARACTER MATRIX 연산 시 `base_params + delta_params` 구조로 분리하고, `delta`가 ±20 범위를 벗어나지 못하도록 클램핑한다. 즉, 캐릭터의 본래 성격은 유지하되 사용자와의 친밀도, 말투 온도만 조금씩 변하는 구조다.

### Q3. 단기 맥락 vs 장기 페르소나 분리
> "단순히 유사 대화를 불러오는 것과 사용자의 장기 성향을 모델링하는 것은 다를 것 같습니다. 단기 대화 맥락과 장기 사용자 페르소나를 어떻게 구분해서 관리하나요?"

**답변:**
두 개의 분리된 저장 구조를 사용한다.

- **단기 맥락 (`session_messages` 테이블)**: 현재 세션의 최근 N턴(약 10턴)만 유지. GPT-4o 컨텍스트 윈도우에 직접 주입. 세션이 끝나면 벡터 임베딩으로 변환해 장기 저장소로 이동 후 삭제.
- **장기 페르소나 (`user_persona` 테이블)**: 세션 종료마다 EMA로 업데이트되는 5축 누적 점수. 이 값이 CHARACTER MATRIX의 입력이 되어 캐릭터 반응 방식을 결정.
- **장기 기억 검색 (`conversation_embeddings`, pgvector)**: 이전 대화 내용을 벡터로 저장해 RAG로 검색. 현재 주제와 유사한 과거 발화를 GPT-4o 시스템 프롬프트에 보조 맥락으로 주입.

요약하면: "어떻게 말하는 사람인가"는 `user_persona`(수치)가, "무슨 얘기를 했었나"는 `conversation_embeddings`(벡터)가 담당한다.

### Q4. 비용과 수익 구조의 지속 가능성
> "GPT-4o 기반이면 사용자 1인당 API 비용이 대화량에 따라 상당히 올라갑니다. 특히 '끊임없이 대화할 수 있는 상대'를 원하는 사용자일수록 API 비용이 폭증하는데, 수익 구조가 지속 가능한가요?"

**답변:**
세 가지 방향으로 비용을 관리한다.

1. **모델 분기**: 5축 분석(analyzer.py)과 CHARACTER MATRIX 연산은 GPT 호출 없이 자체 엔진으로 처리. GPT-4o는 최종 응답 생성에만 사용해 호출 횟수를 최소화.
2. **티어 설계**: 무료 플랜은 일 20턴 제한, 유료 플랜(월정액)은 무제한. 헤비유저가 유료로 자연스럽게 전환되도록 설계.
3. **프롬프트 캐싱 활용**: 시스템 프롬프트(캐릭터 정의, 5축 점수 등)는 매 요청마다 반복되므로 Anthropic/OpenAI의 프롬프트 캐싱을 적용해 반복 토큰 비용을 절감.

캡스톤 MVP 단계에서는 비용 임계치를 초과하면 응답 모델을 GPT-4o-mini로 자동 fallback하는 구조도 검토 중이다.

### Q5. 5축 분석 효과 검증
> "사용자 발화를 5개 축으로 분석한 결과가 실제로 '개인화된 영어 회화 경험'으로 느껴진다는 것을 어떻게 검증할 예정이신가요?"

**답변:**
두 단계로 검증한다.

**1단계 — 기술 정합성 검증 (현재 진행 중):**
`data/dataset.py`의 수작업 라벨 25개와 analyzer.py 출력값의 일치율을 측정. 현재 MVPf는 정확도보다 "스타일 차이를 구분하는가"(캐주얼 vs 격식 분리 여부)에 집중.

**2단계 — 사용자 체감 검증 (베타 테스트 단계):**
- A/B 테스트: 그룹 A는 5축 기반 개인화 응답, 그룹 B는 동일 캐릭터 고정 응답. 세션 지속 시간, 재방문율, "이 응답이 나에게 맞는 느낌이었나요?" 인앱 마이크로 서베이로 비교.
- 지표: 5턴 이상 지속 세션 비율, 1주일 내 재방문율을 핵심 지표로 설정.

"개인화 느낌"은 주관적이기 때문에 수치 정확도보다 행동 지표(더 오래 대화하는가, 다시 오는가)로 검증하는 것이 더 유효하다고 판단하고 있다.

### Q6. 과몰입 / 의존성 리스크
> "실제로 일부 AI 서비스에서 나타났던 과몰입이나 정서적 의존, 망상 같은 문제가 발생할 가능성도 있지 않을까요? 이러한 위험을 방지할 방안이 있을까요?"

**답변:**
Character.AI, Replika 사례를 인지하고 있으며, 설계 단계부터 세 가지 안전장치를 둔다.

1. **학습 도구 포지셔닝 유지**: 캐릭터는 "영어 학습 파트너"로 일관되게 프레이밍. 순수 감정 지지나 로맨틱 관계 시뮬레이션 방향으로의 대화는 시스템 프롬프트 레벨에서 차단.
2. **사용 패턴 피드백**: 주간 사용 리포트를 사용자에게 제공("이번 주 N회 대화, 총 M분"). 일 사용 시간이 임계값(예: 2시간)을 초과하면 "잠깐 쉬어가세요" 넛지를 표시.
3. **AI임을 명시**: 대화 UI 상단에 "AI 캐릭터와 대화 중" 라벨을 항상 노출. 캐릭터가 "나 진짜 사람이야" 류의 응답을 생성하지 못하도록 시스템 프롬프트에 명시적 금지 지시 포함.

정서적 유대감은 학습 지속의 수단이지 목적이 아니라는 점을 제품 전반에서 일관되게 유지하는 것이 핵심이다.

## 3. 프로젝트에서 참고할 것

### 즉시 반영 가능 (설계 단계)

| 이슈 | 참고 방향 |
|------|-----------|
| **콜드 스타트** | 첫 세션에서도 차별점을 체감하게 할 "퀵 온보딩" 설계 필요. 예: 첫 대화 3턴만으로도 캐릭터 변화가 눈에 띄게 설정 |
| **페르소나 제어** | 5축 중 변하는 값(관계 축: Intimacy)과 덜 변하는 값(기본 성격 축: Formality 범위 클램핑)을 분리 설계 |
| **단기/장기 메모리** | Supabase에서 `session_context` 테이블(단기)과 `user_persona` 테이블(장기)을 분리해 설계 |
| **피드백 타이밍** | 문법 교정은 발화 직후가 아니라 대화 턴 끝 또는 세션 종료 시 요약 피드백으로 설계 |

### 중기 로드맵에 반영 (Phase 2~3)

| 이슈 | 참고 방향 |
|------|-----------|
| **비용 구조** | 일일 대화 제한(무료) + 무제한 구독(유료) 모델 명시. GPT-4o는 Sonnet 4.5 수준으로 fallback 가능 여부 검토 |
| **슬랭 필터링** | Reddit 슬랭 RAG에서 학습 목적 태그(casual / professional / academic) 필드 추가해 Humor 축 점수 기반 주입 여부 결정 |
| **과몰입 방지** | 주간 대화 시간 요약 표시, "현실 친구와도 영어 연습해보세요" 류의 넛지 설계 |

### 발표 대응 답변으로 미리 준비할 것

- **"ChatGPT랑 뭐가 달라요?"** → ChatGPT는 매 대화가 독립적, Pally는 대화가 쌓일수록 캐릭터가 변화하는 *관계의 연속성*이 차별점
- **"초기 세션에서 어떻게 붙잡나요?"** → 온보딩 첫 3턴에서 캐릭터 변화를 *과장하여* 보여주는 "데모 모드" 운영 후 실제 EMA 적용으로 전환
- **"과몰입은요?"** → 주간 사용 리포트 + 학습 목표 리마인더로 "학습 도구" 포지셔닝 유지
