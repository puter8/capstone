## 2. AI 제안을 수정하거나 기각한 사례

### 사례 A 
- 상황: 초기 계획은 GPT-4o 단일 호출로 `축 점수 + 답변`을 함께 생성하는 구조였다.
- AI 제안 요약: `structured output` 하나로 점수 추출과 답변 생성을 통합하자는 방향.
- 우리의 판단: 파싱 실패 시 점수 업데이트까지 동시에 깨져서, 재현성과 디버깅 안정성이 낮다고 판단했다.
- 최종 결정: `Rule-Based 분석기 + MATRIX 엔진` 구조로 변경하고, GPT-4o는 답변 생성만 담당하도록 역할을 분리했다.

### 사례 B 
- 상황: 캐릭터 애니메이션 구현 방식을 팀에서 아직 확정하지 못한 상태였다.
- AI 제안 요약: 구현 난이도와 일반적인 사용성을 이유로 Rive 기반 애니메이션을 제안했다.
- 우리의 판단: 우리 프로젝트는 5축 수치가 어떻게 시각 변화로 이어지는지 발표에서 설명 가능해야 해서, 내부 파라미터를 직접 제어할 수 있는 방식이 더 적합하다고 봤다.
- 최종 결정: `Superformula + Canvas2D`를 채택해 축→형태/색/호흡 매핑을 직접 설계했다.

### 사례 C 
- 상황: 초기 초안에는 슬랭 수집을 `Reddit + YouTube API`, RAG를 외부 벡터 DB 중심으로 가져가려는 안이 섞여 있었다.
- AI 제안 요약: 데이터 소스를 다중화하고 벡터 검색도 외부 스택으로 확장하는 방향.
- 우리의 판단: MVP 단계에서 운영 복잡도·비용·동기화 리스크가 커지고, 팀 규모 대비 유지보수 부담이 크다고 판단했다.
- 최종 결정: 슬랭 수집은 `Reddit PRAW` 단독, RAG는 `Supabase pgvector`(기존 DB 내부 벡터 검색)로 단순화했다.

## 3. AI가 틀렸거나 못 믿었던 순간
- 테스트 커버리지 착시: AI가 작성한 변경 설명에는 “pytest 차별화 검증”이 포함되어 있었지만, 실제 실행(`python3 -m pytest -q`) 결과는 `no tests ran in 0.01s`였다. 테스트 파일 재구성 과정에서 pytest 함수가 빠져 검증이 비어 있었음을 확인했다.
- 조치: AI가 적어준 “테스트됨” 문구를 신뢰하지 않고, 반드시 명령 실행 결과로 검증하도록 팀 규칙을 바꿨다.

## 4. AI 없이 우리가 직접 한 것
- 초기 아이디어 정의: “사용자 발화 분석으로 AI 캐릭터 성격을 동적으로 바꾼다”는 제품 콘셉트는 팀이 직접 기획했다.  
  이유: 프로젝트 정체성과 문제 정의는 팀 합의가 필요해 AI에 위임하지 않았다.
- 서비스/캐릭터 디자인 방향: 어떤 감정 경험을 줄지(말투 변화, 캐릭터 변화 체감 포인트)는 팀이 직접 결정했다.  
  이유: 발표 시연에서 전달할 사용자 경험 목표를 팀이 책임져야 했기 때문이다.
- 최종 기술 검토/채택 책임: AI가 만든 문서·코드 초안은 그대로 쓰지 않고 팀이 검토 후 채택/수정/기각을 결정했다.  
  이유: 구현 책임과 발표 책임은 팀에게 있으므로, 최종 의사결정권은 사람에게 두었다.
